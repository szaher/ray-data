## From Batch to Streaming

Traditional batch pipelines read all data, process it, then write results. Streaming pipelines process data continuously as it arrives -- block by block. Ray Data uses streaming execution by default, and this pattern extends naturally to real-time workloads.

## Streaming Execution in Ray Data

Ray Data's `iter_batches()` and `iter_torch_batches()` enable streaming consumption. The pipeline produces blocks on-demand as the consumer pulls them.

<CodeBlock language="python" code={`import ray

# Streaming read + process: only a few blocks in memory at a time
ds = ray.data.read_parquet("s3://my-bucket/events/", parallelism=200)
ds = ds.map_batches(transform_events, batch_size=2048)

# Stream results to a consumer — blocks flow through lazily
for batch in ds.iter_batches(batch_size=4096):
    send_to_downstream(batch)

# For ML training: stream directly to PyTorch
train_ds = ray.data.read_parquet("s3://data/train/")
train_ds = train_ds.map_batches(preprocess, batch_size=1024)

for epoch in range(10):
    for batch in train_ds.iter_torch_batches(batch_size=256):
        loss = model.train_step(batch)  # Blocks stream in as needed`} />

## Continuous Ingestion Pattern

For near-real-time workloads, you can build a loop that continuously reads new data partitions and processes them through your pipeline.

<CodeBlock language="python" code={`import ray
import time
from datetime import datetime, timedelta

def process_latest_partition():
    """Process the latest hourly partition as it lands."""
    current_hour = datetime.utcnow().replace(minute=0, second=0)
    partition_path = current_hour.strftime(
        "s3://my-bucket/events/dt=%Y-%m-%d/hr=%H/"
    )

    ds = ray.data.read_parquet(partition_path)
    ds = ds.map_batches(parse_and_validate, batch_size=2048)
    ds = ds.filter(lambda row: row["is_valid"])
    ds = ds.map_batches(compute_features, batch_size=1024)
    ds.write_parquet(
        current_hour.strftime(
            "s3://my-bucket/features/dt=%Y-%m-%d/hr=%H/"
        )
    )
    return ds.count()

# Continuous ingestion loop
while True:
    rows = process_latest_partition()
    print(f"Processed {rows} rows")
    time.sleep(3600)  # Wait for next hourly partition`} />

## Backpressure and Flow Control

Ray Data automatically applies backpressure when downstream operators are slower than upstream ones. This prevents out-of-memory errors without any configuration.

## Key Takeaways

- Ray Data uses **streaming execution by default** -- data flows block-by-block through operators
- Use **`iter_batches()`** to consume results lazily without loading the full dataset
- Build **continuous ingestion loops** for near-real-time partition processing
- **Backpressure** is automatic: slow operators cause upstream operators to pause, keeping memory bounded
