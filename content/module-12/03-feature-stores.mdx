## The feature store pattern

ML models need features at two moments: training (batch, historical data) and inference (real-time, latest values). A feature store bridges these two needs, and Ray Data works well as the compute layer that feeds it.

## Computing features with Ray Data

Define feature transformations as Ray Data operations. These transforms run at scale across a cluster, then produce outputs in a format the feature store can ingest.

<CodeBlock language="python" code={`import ray
import pandas as pd

def compute_user_features(batch: pd.DataFrame) -> pd.DataFrame:
    """Aggregate user-level features from raw events."""
    features = batch.groupby("user_id").agg(
        total_purchases=("amount", "sum"),
        avg_purchase=("amount", "mean"),
        purchase_count=("amount", "count"),
        last_active=("timestamp", "max"),
    ).reset_index()

    features["days_since_active"] = (
        pd.Timestamp.utcnow() - features["last_active"]
    ).dt.days
    return features

# Scale feature computation across the cluster
ds = ray.data.read_parquet("s3://data-lake/events/purchases/")
features_ds = ds.map_batches(
    compute_user_features, batch_size=10_000
)

# Write to the offline store location
features_ds.write_parquet("s3://feature-store/offline/user_features/")`} />

## Feast integration

Feast is a popular open-source feature store. After Ray Data computes features and writes them to storage, Feast manages versioning, retrieval, and online/offline consistency.

<CodeBlock language="python" code={`# feature_repo/features.py — Feast feature definition
from feast import Entity, FeatureView, FileSource, Field
from feast.types import Float32, Int64
from datetime import timedelta

user = Entity(name="user_id", join_keys=["user_id"])

user_features_source = FileSource(
    path="s3://feature-store/offline/user_features/",
    timestamp_field="last_active",
)

user_features_view = FeatureView(
    name="user_features",
    entities=[user],
    ttl=timedelta(days=1),
    schema=[
        Field(name="total_purchases", dtype=Float32),
        Field(name="avg_purchase", dtype=Float32),
        Field(name="purchase_count", dtype=Int64),
        Field(name="days_since_active", dtype=Int64),
    ],
    source=user_features_source,
)`} />

<CodeBlock language="python" code={`# Retrieve features for training (offline) and serving (online)
from feast import FeatureStore

store = FeatureStore(repo_path="./feature_repo")

# Offline: get historical features for training
training_data = store.get_historical_features(
    entity_df=entity_df,  # DataFrame with user_id + timestamps
    features=[
        "user_features:total_purchases",
        "user_features:avg_purchase",
        "user_features:days_since_active",
    ],
).to_df()

# Online: get latest features for real-time inference
online_features = store.get_online_features(
    entity_rows=[{"user_id": "u_12345"}],
    features=["user_features:total_purchases"],
).to_dict()`} />

## Key takeaways

- Ray Data handles computation, the feature store handles storage and retrieval. Each does what it does best.
- Write Ray Data outputs to the offline store (Parquet/BigQuery), then materialize to the online store (Redis/DynamoDB).
- Feast provides versioned feature definitions, point-in-time-correct joins for training, and low-latency online lookups.
- This pattern gives you a single source of truth for features used in both training and serving.
