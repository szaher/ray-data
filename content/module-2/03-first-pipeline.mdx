## From zero to pipeline in 10 lines

You know how to run functions on Ray workers. Now, process **data**. Ray Data gives you a pipeline API that reads, transforms, filters, and writes data, all distributed automatically.

Here is a complete pipeline:

<CodeBlock code={`import ray
ray.init()
ds = ray.data.read_csv("s3://anonymous@ray-example-data/iris.csv")
ds = ds.map(lambda row: {**row, "sepal_area": row["sepal length (cm)"] * row["sepal width (cm)"]})
ds = ds.filter(lambda row: row["sepal_area"] > 15)
ds = ds.select_columns(["sepal_area", "target"])
print(f"Rows after filter: {ds.count()}")
ds.show(5)
ds.write_parquet("/tmp/iris_filtered")
print("Done! Parquet files written to /tmp/iris_filtered/")`} language="python" filename="first_pipeline.py" />

That's a real, distributed data pipeline. Here's what each step does.

## Step 1: Read data

Ray Data can read from CSV, Parquet, JSON, and many other formats. It can also create datasets from Python objects:

<CodeBlock code={`# From a CSV file
ds = ray.data.read_csv("data.csv")
# From a Parquet file (faster, columnar)
ds = ray.data.read_parquet("data.parquet")
# From a Python list (great for experiments)
ds = ray.data.from_items([
    {"name": "Alice", "age": 30},
    {"name": "Bob", "age": 25},
    {"name": "Carol", "age": 35},
])
# From S3 (no credentials needed for public buckets)
ds = ray.data.read_csv("s3://anonymous@ray-example-data/iris.csv")`} language="python" filename="read_data.py" />

`read_csv()` returns a **Ray Dataset**, not a pandas DataFrame. The data is split into **blocks** that can be processed in parallel across workers.

## Step 2: Transform with .map()

`.map()` applies a function to every row. Each row is a Python dictionary where keys are column names:

<CodeBlock code={`ds = ray.data.from_items([
    {"name": "Alice", "score": 85},
    {"name": "Bob", "score": 92},
    {"name": "Carol", "score": 78},
])
# Add a "passed" column based on score
ds = ds.map(lambda row: {**row, "passed": row["score"] >= 80})
ds.show()
# {'name': 'Alice', 'score': 85, 'passed': True}
# {'name': 'Bob',   'score': 92, 'passed': True}
# {'name': 'Carol', 'score': 78, 'passed': False}`} language="python" filename="map_transform.py" />

The `{**row, "passed": ...}` pattern copies the existing row and adds a new field. This is the standard way to add columns in Ray Data.

## Step 3: Filter with .filter()

`.filter()` keeps only the rows where your function returns `True`:

<CodeBlock code={`# Keep only rows where the student passed
ds_passed = ds.filter(lambda row: row["passed"] == True)
ds_passed.show()
# {'name': 'Alice', 'score': 85, 'passed': True}
# {'name': 'Bob',   'score': 92, 'passed': True}`} language="python" filename="filter_rows.py" />

You can chain `.map()` and `.filter()` as many times as you need. Each call adds a step to the pipeline.

## Step 4: Inspect with .show() and .count()

Use these methods to check your data at any point:

<CodeBlock code={`# Show the first N rows (default 20)
ds.show(5)
# Count total rows
print(ds.count())  # 3
# Take rows as a Python list of dicts
rows = ds.take(2)
print(rows)  # [{'name': 'Alice', ...}, {'name': 'Bob', ...}]
# Get the schema
print(ds.schema())
# Column  Type
# ------  ----
# name    string
# score   int64
# passed  bool`} language="python" filename="inspect_data.py" />

## Step 5: Write results

Save your processed data to disk. Parquet is the recommended format because it's columnar, compressed, and fast:

<CodeBlock code={`# Write to Parquet (recommended)
ds.write_parquet("/tmp/output/")
# Write to CSV
ds.write_csv("/tmp/output_csv/")
# Write to JSON
ds.write_json("/tmp/output_json/")`} language="python" filename="write_data.py" />

Ray Data commonly writes a directory of output files. The file count depends on the number of dataset blocks and writer configuration. This is normal for distributed systems — inspect the output directory to see the actual layout.

Data flows left to right, one block at a time. While block 1 is being filtered, block 2 can still be in the map stage. This **pipelining** keeps all your CPU cores busy.

## Key concept: lazy execution

Ray Data pipelines are **lazy**. When you call `.map()` or `.filter()`, nothing happens yet. Ray builds a plan but doesn't process any data. Execution starts only when you call a **consuming operation**:

- `.show()` prints rows (triggers execution)
- `.count()` counts rows (triggers execution)
- `.take(n)` returns n rows as a list (triggers execution)
- `.write_parquet()` writes to disk (triggers execution)

<CodeBlock code={`import ray
ray.init()
ds = ray.data.from_items([{"x": i} for i in range(1000)])
# These lines build a plan — nothing runs yet
ds = ds.map(lambda row: {**row, "y": row["x"] * 2})
ds = ds.filter(lambda row: row["y"] > 500)
# THIS triggers execution of the entire pipeline
ds.show(5)`} language="python" filename="lazy_execution.py" />

To force execution without consuming the result (for example, to cache data in memory), use `.materialize()`:

<CodeBlock code={`# Force execution and keep the result in Ray's object store
ds = ds.materialize()
# Now .count() and .show() are instant — data is already computed
print(ds.count())
ds.show(5)`} language="python" filename="materialize.py" />

## Putting it all together

Here is the full pipeline from the top of this lesson, with comments explaining each step:

<CodeBlock code={`import ray
# 1. Start a local Ray cluster
ray.init()
# 2. Read CSV data (from a public S3 bucket)
ds = ray.data.read_csv("s3://anonymous@ray-example-data/iris.csv")
# 3. Add a computed column
ds = ds.map(lambda row: {
    **row,
    "sepal_area": row["sepal length (cm)"] * row["sepal width (cm)"]
})
# 4. Keep only large petals
ds = ds.filter(lambda row: row["sepal_area"] > 15)
# 5. Select just the columns we need
ds = ds.select_columns(["sepal_area", "target"])
# 6. Inspect the result
print(f"Rows after filter: {ds.count()}")
ds.show(5)
# 7. Save to Parquet
ds.write_parquet("/tmp/iris_filtered")
# 8. Clean up
ray.shutdown()`} language="python" filename="full_pipeline.py" />

You've just built a distributed data pipeline. The same transformation logic runs on your laptop and on a cluster. Cluster deployment also requires compatible dependencies, storage access, credentials, and resource configuration — but the data processing code itself stays the same.
