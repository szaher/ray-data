## From functions to tasks

A Ray **task** is a function that runs in a separate worker process. You create one by adding the `@ray.remote` decorator, then calling it with `.remote()` instead of `()`.

<CodeBlock language="python" code={`import ray

ray.init()

# Regular Python function
def square(x):
    return x * x

# Ray task — runs in a worker process
@ray.remote
def square_remote(x):
    return x * x

# Calling a task returns an ObjectRef, not the value
ref = square_remote.remote(4)   # Returns immediately
result = ray.get(ref)           # Blocks until done → 16
`} />

## The execution model

When you call `.remote()`, three things happen: the function and its arguments are serialized, the task is scheduled onto an available worker, and an `ObjectRef` is returned immediately. Your driver script keeps running. It does not wait.

## Parallel execution

Tasks become useful when you launch many at once. Submit all your work, then collect results:

<CodeBlock language="python" code={`import ray
import time

@ray.remote
def slow_square(x):
    time.sleep(1)  # Simulate expensive work
    return x * x

# WRONG: Sequential — takes 10 seconds
results = []
for i in range(10):
    ref = slow_square.remote(i)
    results.append(ray.get(ref))  # Blocks each iteration!

# RIGHT: Parallel — takes ~1 second (with 10+ CPUs)
refs = [slow_square.remote(i) for i in range(10)]
results = ray.get(refs)  # Block once, get all results
`} />

## Passing arguments and dependencies

Tasks accept any serializable Python object. You can also pass `ObjectRef`s directly. Ray automatically resolves them before the task runs, creating **task dependencies** without ever fetching data to the driver.

<CodeBlock language="python" code={`@ray.remote
def load_data(path):
    import pandas as pd
    return pd.read_csv(path)

@ray.remote
def process(df):
    return df.describe()

# Pipeline: load_data → process (no data hits the driver)
data_ref = load_data.remote("data.csv")
result_ref = process.remote(data_ref)  # Pass the ref directly
stats = ray.get(result_ref)
`} />

## Resource requirements

You can specify how many CPUs, GPUs, or custom resources a task needs:

<CodeBlock language="python" code={`@ray.remote(num_cpus=4, num_gpus=1)
def train_model(data):
    # This task reserves 4 CPUs and 1 GPU
    ...

# Override resources at call time
train_model.options(num_cpus=2).remote(data)
`} />

Ray's scheduler will only place the task on a node that has the requested resources available. If no node qualifies, the task waits in the queue.
