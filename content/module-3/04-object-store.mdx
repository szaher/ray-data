## The ObjectRef

Every `.remote()` call returns an **ObjectRef**, a pointer to a value that may not exist yet. The actual data lives in Ray's distributed **object store**, not in your driver process.

<CodeBlock language="python" code={`import ray
import numpy as np

ray.init()

@ray.remote
def generate_data():
    return np.random.rand(1_000_000)

ref = generate_data.remote()
print(type(ref))   # <class 'ray.ObjectRef'>
print(ref)          # ObjectRef(00ffffffffffffffffffffff...)

data = ray.get(ref)  # Fetch the actual numpy array
print(data.shape)    # (1000000,)
`} />

## How the object store works

Each node runs an **object store** (built on Apache Arrow's Plasma). It uses shared memory so multiple worker processes on the same node can access the same data without copying it.

## ray.put(): explicit storage

When you pass a large object to many tasks, the driver serializes it for **each** call. `ray.put()` stores it once so you can pass the ref instead:

<CodeBlock language="python" code={`import numpy as np

big_array = np.random.rand(10_000_000)

# BAD: Serializes big_array 100 times
refs = [process.remote(big_array) for _ in range(100)]

# GOOD: Serialize once, share the ref
big_ref = ray.put(big_array)
refs = [process.remote(big_ref) for _ in range(100)]
`} />

For a 100 MB array sent to 100 tasks, this takes you from **10 GB** of serialization down to **100 MB**.

## Zero-copy reads

When a worker on the same node needs an object, it reads directly from shared memory via memory mapping. No data is copied into the worker's heap. This is **zero-copy deserialization**, and it works with NumPy arrays and Arrow tables out of the box.

## Object lifetime and spilling

Objects stay in memory as long as an `ObjectRef` references them. When the object store fills up, Ray **spills** objects to disk automatically and reloads them on demand.

<CodeBlock language="python" code={`# Objects are reference-counted
ref = ray.put(large_data)    # Stored in object store
del ref                       # Reference dropped → eligible for eviction

# ray.get() on a spilled object reloads it transparently
result = ray.get(some_old_ref)  # May trigger disk read

# Check object store memory
print(ray.cluster_resources())
# {'object_store_memory': 8589934592, ...}
`} />

## Best practices

Use `ray.put()` when passing the same large object to many tasks. Pass `ObjectRef`s between tasks to build pipelines without pulling data to the driver. Avoid calling `ray.get()` too early; it blocks and defeats the purpose of async execution. Let Ray manage data locality as long as possible.
