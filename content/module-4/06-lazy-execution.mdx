## Lazy by default

When you chain transformations in Ray Data, **nothing executes**. Each call to `map_batches`, `filter`, or `map` adds a step to a logical plan. Execution is deferred until you consume the results.

## Why lazy execution?

Lazy execution enables two optimizations:

1. **Operator fusion** -- Ray Data can merge adjacent map steps into a single pass over the data, reducing serialization overhead.
2. **Streaming execution** -- data flows through the pipeline block-by-block instead of materializing every intermediate result in memory.

<CodeBlock language="python" code={`import ray

# None of these lines trigger computation
ds = ray.data.read_parquet("s3://bucket/events/")      # plan: read
ds = ds.map_batches(parse_json_fields)                  # plan: read -> map
ds = ds.filter(lambda row: row["status"] == "active")   # plan: read -> map -> filter
ds = ds.map_batches(extract_features)                   # plan: read -> map -> filter -> map

# Nothing has run yet. ds is just a plan.
print(ds)
# Dataset(
#   num_rows=?,
#   schema={...}
# )

# THIS triggers execution:
ds.show(5)`} />

## What triggers execution?

These operations cause the pipeline to run:

| Operation | What It Does |
|-----------|-------------|
| `show(n)` | Prints first n rows |
| `take(n)` | Returns first n rows as list |
| `count()` | Returns total row count |
| `iter_rows()` | Streams rows one at a time |
| `iter_batches()` | Streams batches |
| `write_parquet()` | Writes results to files |
| `write_csv()` | Writes results to CSV |
| `materialize()` | Forces full execution, pins in memory |

## materialize(): pinning results

By default, once data flows through a consumer, the intermediate blocks are freed. If you need to iterate over the same dataset multiple times (for example, multiple training epochs), use `materialize()` to keep results in memory.

<CodeBlock language="python" code={`# Without materialize — re-reads and re-computes every epoch
ds = ray.data.read_parquet("s3://bucket/train/")
ds = ds.map_batches(preprocess)

for epoch in range(5):
    for batch in ds.iter_batches(batch_size=128):  # re-executes pipeline each time
        train(batch)

# With materialize — compute once, iterate many times
ds = ray.data.read_parquet("s3://bucket/train/")
ds = ds.map_batches(preprocess)
ds = ds.materialize()  # executes pipeline, pins blocks in object store

for epoch in range(5):
    for batch in ds.iter_batches(batch_size=128):  # reads from memory
        train(batch)`} />

## When to materialize

Materialize when you will consume the same dataset multiple times (multi-epoch training, multiple aggregations). Skip it when data is consumed once (single-pass inference, write-and-forget ETL). Streaming execution is more memory-efficient for single-pass workloads.

<CodeBlock language="python" code={`# Good use of materialize: multi-epoch training
train_ds = ray.data.read_parquet("train/").map_batches(prep).materialize()
val_ds = ray.data.read_parquet("val/").map_batches(prep).materialize()

for epoch in range(10):
    train_loss = train_epoch(train_ds)
    val_loss = eval_epoch(val_ds)
    print(f"Epoch {epoch}: train={train_loss:.3f} val={val_loss:.3f}")

# Good use of streaming: one-shot batch inference
ds = ray.data.read_images("s3://bucket/images/")
# gpu-example: lazy-inference
ds = ds.map_batches(Predictor, num_gpus=1, compute=ray.data.ActorPoolStrategy(size=2))
ds.write_parquet("s3://bucket/predictions/")  # stream through, no materialize needed`} />
