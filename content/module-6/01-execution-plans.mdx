## From logical plan to physical execution

When you chain operations on a Ray Dataset, nothing executes immediately. Ray Data builds a logical plan, a DAG of operators representing your intent. Only when you trigger an action (like `take()`, `write`, or `iter_batches()`) does the planner convert it into an optimized physical plan.

## Inspecting plans with `.explain()`

You can inspect what Ray Data will actually do before triggering execution.

<CodeBlock language="python" code={`import ray

ds = (
    ray.data.read_parquet("s3://bucket/data/")
    .map(lambda row: {**row, "name": row["name"].lower()})
    .filter(lambda row: row["age"] > 18)
    .map_batches(lambda batch: batch)
)

# Print the logical and physical plans
ds.explain()
# == Logical Plan ==
# ReadParquet -> Map(fn) -> Filter(fn) -> MapBatches(fn)
#
# == Optimized Physical Plan ==
# ReadParquet -> MapBatches(fn) [fused: Map, Filter, MapBatches]`} />

## Operator fusion rules

The optimizer fuses operators when these conditions are met:

- One-to-one operators like `map`, `filter`, and `map_batches` can be fused together into a single stage.
- A read operator followed by map-like operators can be fused so transformation happens during reading, avoiding a full materialization pass.
- Fusion breaks at shuffle boundaries (`repartition`, `random_shuffle`, `sort`, `groupby`) because data must be fully materialized and redistributed.

<CodeBlock language="python" code={`# These three map-like ops will be fused into ONE physical stage
ds = (
    ray.data.read_parquet("data/")
    .map(parse_record)           # ─┐
    .filter(is_valid)            # ─┤ fused into a single task
    .map_batches(normalize)      # ─┘
)

# Adding a shuffle BREAKS the fusion chain
ds = (
    ray.data.read_parquet("data/")
    .map(parse_record)           # ─┐ fused stage 1
    .filter(is_valid)            # ─┘
    .repartition(100)            # ← shuffle barrier
    .map_batches(normalize)      # ─── fused stage 2
)`} />

## Controlling fusion

You can disable fusion or tune plan behavior via `ExecutionOptions`.

<CodeBlock language="python" code={`ctx = ray.data.DataContext.get_current()

# Disable operator fusion entirely (useful for debugging)
ctx.execution_options.preserve_order = True

# Force materialization between stages for debugging
ds = ray.data.read_parquet("data/").materialize()
ds_filtered = ds.filter(is_valid)  # starts a new plan`} />

Understanding execution plans helps you reason about performance. If two stages aren't fusing when you expect them to, `explain()` will reveal why and guide your optimization.
