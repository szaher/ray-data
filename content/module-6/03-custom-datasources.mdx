## When you need a custom datasource

Ray Data has built-in connectors for Parquet, CSV, JSON, images, and more. But when your data lives in a proprietary system (a custom database, a message queue, an internal API), you need to implement your own `Datasource`.

## Implementing a read datasource

The core interface requires implementing `get_read_tasks()`, which returns a list of `ReadTask` objects. Each `ReadTask` is a zero-argument callable that produces an iterator of Arrow tables or pandas DataFrames.

<CodeBlock language="python" code={`import ray
from ray.data.datasource import Datasource, ReadTask
from ray.data.block import BlockMetadata
import pyarrow as pa

class MyAPIDatasource(Datasource):
    def __init__(self, base_url: str, partitions: int = 4):
        self.base_url = base_url
        self.partitions = partitions

    def get_read_tasks(self, parallelism: int) -> list[ReadTask]:
        tasks = []
        for i in range(self.partitions):
            # Metadata helps the scheduler plan memory
            metadata = BlockMetadata(
                num_rows=1000,                  # estimated
                size_bytes=1000 * 256,          # estimated bytes
                schema=pa.schema([
                    ("id", pa.int64()),
                    ("value", pa.float64()),
                ]),
                input_files=None,
                exec_stats=None,
            )
            # Capture loop variable
            url = f"{self.base_url}/partition/{i}"
            read_fn = self._make_read_fn(url)
            tasks.append(ReadTask(read_fn, metadata))
        return tasks

    def _make_read_fn(self, url):
        def read() -> iter:
            import requests
            resp = requests.get(url).json()
            table = pa.table({
                "id": [r["id"] for r in resp],
                "value": [r["value"] for r in resp],
            })
            return [table]   # Return list of Arrow tables
        return read`} />

## Using your custom datasource

<CodeBlock language="python" code={`# Read from your custom source
ds = ray.data.read_datasource(
    MyAPIDatasource(base_url="https://api.example.com", partitions=8)
)

# Use it like any other Dataset
ds = ds.filter(lambda row: row["value"] > 0.5)
print(ds.take(5))`} />

## Implementing a write datasource

To support writing, override `do_write()`. Ray Data calls this with blocks of data to persist.

<CodeBlock language="python" code={`class MyAPIWriteDatasource(Datasource):
    def __init__(self, endpoint: str):
        self.endpoint = endpoint

    def do_write(self, blocks, metadata, ray_remote_args,
                 write_args) -> list:
        import requests

        def write_block(block: pa.Table):
            payload = block.to_pydict()
            requests.post(self.endpoint, json=payload)

        # Write each block in parallel via Ray tasks
        write_tasks = [
            ray.remote(write_block).remote(block)
            for block in blocks
        ]
        ray.get(write_tasks)
        return []

# Write a dataset to your custom sink
ds.write_datasource(MyAPIWriteDatasource("https://api.example.com/ingest"))`} />

## Design considerations

- Partition wisely. Return one `ReadTask` per natural data partition (files, shards, API pages). More tasks means more parallelism.
- Estimate metadata accurately. Bad size estimates lead to poor backpressure decisions, causing spilling or underutilization.
- Return Arrow tables. They integrate most efficiently with Ray Data's block format and avoid extra serialization.
- Handle retries in your read function. Each `ReadTask` callable may be re-executed on failure, so make reads idempotent.
