## Failures are inevitable at scale

When processing terabytes across dozens of machines, node failures, preemptions, and OOM kills are normal. Ray Data recovers transparently using task-level retries and lineage-based reconstruction.

## Block-level retries

Ray Data retries at the task level, not the job level. Each task processes one or more blocks. By default, Ray retries a task up to three times when its worker process or node dies. Application exceptions are **not** retried unless you explicitly enable `retry_exceptions`.

<CodeBlock language="python" code={`import ray
from ray.data import ActorPoolStrategy

ds = ray.data.read_parquet("s3://bucket/data/")

# Default: retries worker/node crashes up to 3 times.
# Application exceptions are NOT retried.
ds = ds.map_batches(stable_transform, batch_size=1024)

# To also retry application exceptions, pass ray_remote_args:
ds = ds.map_batches(
    flaky_transform,
    batch_size=1024,
    ray_remote_args={"max_retries": 5, "retry_exceptions": True},
)`} />

## Lineage reconstruction

When a worker holding computed blocks dies, those blocks are lost from the object store. Ray Data reconstructs them using lineage: the recorded chain of operations and inputs that produced each block.

## In-memory materialization with materialize()

`materialize()` executes the dataset and pins the returned blocks in Ray's object-store memory for reuse during the running cluster. It is not a durable checkpoint. For recovery across node failures or job restarts, write intermediate results to durable storage.

<CodeBlock language="python" code={`import ray

# Without materialization: failure at stage 3 re-executes from stage 1
ds = (
    ray.data.read_parquet("s3://bucket/raw/")
    .map_batches(expensive_parse)
    .map_batches(expensive_transform)    # if this fails...
    .map_batches(expensive_inference)    # ...recompute from read
)

# With materialization: re-execution starts here, not from read
ds = ray.data.read_parquet("s3://bucket/raw/")
ds = ds.map_batches(expensive_parse).materialize()    # pin in memory
ds = ds.map_batches(expensive_transform).materialize() # pin in memory
ds = ds.map_batches(expensive_inference)  # if this fails, upstream blocks are already in memory`} />

## Configuring fault tolerance

<CodeBlock language="python" code={`import ray

# Write intermediate results to durable storage for long jobs
ds = ray.data.read_parquet("s3://input/")
ds = ds.map_batches(stage_1)
ds.write_parquet("s3://checkpoint/stage1/")  # durable checkpoint

# Resume from durable checkpoint if the job restarts
ds = ray.data.read_parquet("s3://checkpoint/stage1/")
ds = ds.map_batches(stage_2)
ds.write_parquet("s3://output/")`} />

## Best practices

- **Understand retry defaults.** By default, Ray retries a task up to three times when its worker process or node dies. Set `max_retries` in `ray_remote_args` to override that limit. Application exceptions are not retried unless `retry_exceptions` is enabled.
- **Keep transforms idempotent.** Retries re-execute your function, so avoid side effects that can't be repeated safely.
- **Materialize before expensive stages.** Use `materialize()` to pin results in object-store memory so upstream stages are not re-executed while the cluster is running.
- **Use durable storage for recovery.** Write intermediate results to S3/GCS with `write_parquet()` — this is the only mechanism that survives node failures or job restarts.
- **Use durable sources.** Lineage reconstruction re-reads inputs, so sources like S3 or GCS are safer than ephemeral streams.
- **Monitor retries.** Excessive retries signal a deeper problem (OOM, bad data) rather than transient failures.
