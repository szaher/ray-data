## Failures are inevitable at scale

When processing terabytes across dozens of machines, node failures, preemptions, and OOM kills are normal. Ray Data recovers transparently using task-level retries and lineage-based reconstruction.

## Block-level retries

Ray Data retries at the task level, not the job level. Each task processes one or more blocks. When a task fails (worker crash, OOM, application error), the executor retries just that task on a different worker.

<CodeBlock language="python" code={`import ray

ctx = ray.data.DataContext.get_current()

# Configure max retries per task (default is 3)
ctx.max_task_retries = 5

# Application-level exceptions can also trigger retries
ds = ray.data.read_parquet("s3://bucket/data/").map_batches(
    flaky_transform,
    batch_size=1024,
    # Ray retries this task up to max_task_retries times
    # if the worker crashes or raises an exception
)`} />

## Lineage reconstruction

When a worker holding computed blocks dies, those blocks are lost from the object store. Ray Data reconstructs them using lineage: the recorded chain of operations and inputs that produced each block.

## Checkpointing with materialize()

Long pipelines can have deep lineage chains. If a block near the end is lost, recomputation might cascade all the way back to the source. Use `materialize()` to create checkpoints that truncate lineage.

<CodeBlock language="python" code={`import ray

# Without checkpointing: failure at stage 3 recomputes from stage 1
ds = (
    ray.data.read_parquet("s3://bucket/raw/")
    .map_batches(expensive_parse)
    .map_batches(expensive_transform)    # if this fails...
    .map_batches(expensive_inference)    # ...recompute from read
)

# With checkpointing: failure at stage 3 only recomputes from checkpoint
ds = ray.data.read_parquet("s3://bucket/raw/")
ds = ds.map_batches(expensive_parse).materialize()    # checkpoint!
ds = ds.map_batches(expensive_transform).materialize() # checkpoint!
ds = ds.map_batches(expensive_inference)  # failure only re-runs this`} />

## Configuring fault tolerance

<CodeBlock language="python" code={`import ray

ctx = ray.data.DataContext.get_current()

# Number of times to retry failed tasks
ctx.max_task_retries = 3

# For spot instances: enable Ray's built-in node fault tolerance
ray.init(
    runtime_env={"env_vars": {"RAY_BACKEND_LOG_LEVEL": "info"}},
)

# Write intermediate results to durable storage for long jobs
ds = ray.data.read_parquet("s3://input/")
ds = ds.map_batches(stage_1)
ds.write_parquet("s3://checkpoint/stage1/")  # durable checkpoint

# Resume from durable checkpoint if the job restarts
ds = ray.data.read_parquet("s3://checkpoint/stage1/")
ds = ds.map_batches(stage_2)
ds.write_parquet("s3://output/")`} />

## Best practices

- Keep transforms idempotent. Retries re-execute your function, so avoid side effects that can't be repeated safely.
- Checkpoint before expensive stages. Use `materialize()` or write to durable storage to limit recomputation.
- Use durable sources. Lineage reconstruction re-reads inputs, so sources like S3 or GCS are safer than ephemeral streams.
- Monitor retries. Excessive retries signal a deeper problem (OOM, bad data) rather than transient failures.
