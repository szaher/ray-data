## The cost of shuffles

Most Ray Data operations are pipelined: data flows through map and filter stages without full materialization. Some operations, though, require shuffling, which redistributes data across all workers. Shuffles are expensive because they break the streaming model.

## Repartition

`repartition()` redistributes data into a specified number of blocks. Use it to fix skew (some blocks much larger than others) or to control parallelism for downstream stages.

<CodeBlock language="python" code={`import ray

ds = ray.data.read_parquet("s3://bucket/data/")  # may produce uneven blocks

# Repartition into 100 evenly-sized blocks
ds = ds.repartition(100)

# Shuffle=False avoids an all-to-all exchange by splitting/merging locally
# (faster but doesn't fix cross-node skew)
ds = ds.repartition(100, shuffle=False)`} />

## Sorting

`sort()` performs a distributed sort using sampling-based range partitioning.

<CodeBlock language="python" code={`# Sort by a single column
ds = ray.data.read_parquet("data/").sort("timestamp")

# Sort descending
ds = ds.sort("score", descending=True)

# Multi-column sort
ds = ds.sort(["category", "score"], descending=[False, True])`} />

## GroupBy and aggregations

`groupby()` shuffles data so all rows with the same key are colocated, then applies aggregations or arbitrary group functions.

<CodeBlock language="python" code={`import ray
from ray.data.aggregate import Count, Mean, Max, Sum

ds = ray.data.read_parquet("s3://bucket/events/")

# Built-in aggregations
result = ds.groupby("user_id").aggregate(
    Count(),
    Mean("duration"),
    Max("score"),
    Sum("revenue"),
)
result.show(5)

# Custom group transformation with map_groups
def normalize_group(group):
    """Normalize scores within each group to [0, 1]."""
    scores = group["score"]
    min_s, max_s = scores.min(), scores.max()
    group["score_norm"] = (scores - min_s) / (max_s - min_s + 1e-9)
    return group

ds_normed = ds.groupby("category").map_groups(
    normalize_group, batch_format="pandas"
)`} />

## Random shuffle

`random_shuffle()` randomizes row order across the entire dataset. This is essential for ML training to avoid ordering bias.

<CodeBlock language="python" code={`# Full global shuffle
ds = ray.data.read_parquet("training_data/").random_shuffle()

# Seeded shuffle for reproducibility
ds = ds.random_shuffle(seed=42)

# For ML training, local shuffle within iter_batches is cheaper
# and often sufficient (avoids a full all-to-all exchange)
for batch in ds.iter_batches(
    batch_size=256,
    local_shuffle_buffer_size=1000,  # shuffle within a buffer
):
    train_step(batch)`} />

## When to use each operation

| Operation | Use Case | Cost |
|---|---|---|
| `repartition(n)` | Fix block skew, control parallelism | High (shuffle) |
| `repartition(n, shuffle=False)` | Adjust block count without full shuffle | Low |
| `sort(key)` | Ordered output, range queries | High |
| `groupby(key).aggregate(...)` | Group-level statistics | High |
| `groupby(key).map_groups(fn)` | Custom per-group transforms | High |
| `random_shuffle()` | Global randomization for training | High |
| `iter_batches(local_shuffle_buffer_size=N)` | Approximate shuffle during training | Low |

Avoid shuffles when possible. Use `local_shuffle_buffer_size` instead of `random_shuffle()` when approximate randomness suffices, and prefer `repartition(shuffle=False)` when you only need to adjust block counts.
