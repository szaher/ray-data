## Profiling with ds.stats()

<CodeBlock language="python" code={`import ray, time

ds = ray.data.read_parquet("s3://data/events/", parallelism=200)
ds = ds.map_batches(lambda b: b, batch_size=4096)

start = time.perf_counter()
result = ds.materialize()
elapsed = time.perf_counter() - start

print(f"Total: {elapsed:.1f}s")
print(result.stats())
# ReadParquet: 200 blocks, 12.3s, 450 MB/s
# MapBatches: 200 blocks, 8.1s, 680 MB/s
`} />

## Benchmark framework

<CodeBlock language="python" code={`def benchmark(name, pipeline_fn, runs=3):
    times = []
    for _ in range(runs):
        start = time.perf_counter()
        pipeline_fn().materialize()
        times.append(time.perf_counter() - start)
    avg, best = sum(times) / len(times), min(times)
    print(f"{name}: avg={avg:.1f}s best={best:.1f}s")

for bs in [1024, 4096, 8192]:
    benchmark(f"batch_size={bs}",
              lambda bs=bs: ray.data.read_parquet("s3://data/events/")
                  .map_batches(lambda b: b, batch_size=bs))
`} />

## When to use Ray Data vs alternatives

Ray Data works best for GPU-heavy ML pipelines and mixed CPU/GPU workloads, where streaming execution, native GPU actors, and Arrow zero-copy give it an edge. Spark works best for SQL-heavy analytics and large shuffle joins.

## Common bottleneck fixes

1. **Slow reads**: increase `parallelism` to create more blocks
2. **CPU starvation**: increase `concurrency` or optimize the transform
3. **GPU idle**: upstream CPU is too slow; add CPU workers
4. **OOM**: reduce `batch_size` or lower `target_max_block_size`

Always start with `ds.stats()`, find the slowest operator, and tune that stage first.
