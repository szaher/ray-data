## Two frameworks, different origins

Apache Spark and Ray were built to solve different problems. Spark (2009, UC Berkeley AMPLab) targeted large-scale data processing: MapReduce done right. Ray (2017, UC Berkeley RISELab) targeted distributed Python applications, especially reinforcement learning and ML workloads. Their architectural differences explain when each one shines.

## The driver-worker model

Both frameworks follow a driver-worker pattern, but the internals diverge.

Spark runs on the JVM. The driver creates a SparkContext that communicates with a cluster manager (YARN, Mesos, or Kubernetes) to launch executor JVM processes. Each executor runs tasks as threads within its JVM. This works well for JVM-native workloads but introduces overhead for Python: PySpark tasks must serialize data between the JVM and a Python subprocess via a socket.

Ray is Python-native. The driver connects to the Global Control Store (GCS), which coordinates raylets (one per node). Each raylet manages local worker processes (actual Python processes) and a shared-memory object store. There is no JVM in the critical path.

This is the deepest architectural difference. It affects everything from latency to resource utilization.

Spark's Catalyst optimizer analyzes your logical plan, applies optimization rules (predicate pushdown, column pruning, join reordering), and produces a physical plan divided into stages. Within a stage, tasks run in parallel. Between stages, a shuffle barrier forces all tasks to complete before the next stage begins. This simplifies fault tolerance (just replay a stage) but introduces latency and forces materialization of intermediate data.

Ray Data schedules tasks in a streaming fashion. Block 1 can be read, mapped, and written while Block 2 is still being read. There are no global barriers unless you explicitly force a materialization (like a sort or groupby). This means lower memory footprint and better pipeline utilization, but the scheduler does more work at runtime.

## Memory models

Memory management is where Spark and Ray make fundamentally different tradeoffs.

Spark's JVM memory layout:
- Execution memory for shuffles, joins, sorts, aggregations
- Storage memory for cached RDDs/DataFrames (managed by a unified memory manager that can borrow between execution and storage)
- Off-heap memory (optional) for Tungsten binary storage that avoids GC pressure
- PySpark adds another layer: data must cross the JVM-Python boundary via Arrow serialization

Ray's shared-memory object store:
- Per-node shared-memory object store
- Each node has a shared-memory region (default 30% of RAM)
- Workers on the same node access objects via memory-mapped files — zero-copy for NumPy arrays; other objects are deserialized
- No JVM, no garbage collection pauses, no Tungsten encoding
- Data is stored in Arrow columnar format natively

## The optimizer question

Spark's Catalyst optimizer is arguably its greatest engineering achievement. It applies dozens of optimization rules to your query plan before a single byte of data moves: predicate pushdown, constant folding, join reordering, column pruning. If you write `df.filter(...).select(...)`, Catalyst may push the filter below the select to reduce data early.

Ray Data has no equivalent general-purpose optimizer. It applies some basic optimizations (operator fusion for adjacent map operations, read-compute overlap), but it relies on you to write efficient pipelines. This is a genuine tradeoff. Spark's optimizer handles naive queries gracefully. Ray Data rewards deliberate pipeline design and sometimes requires it.

## When JVM overhead matters

The JVM is both Spark's strength and its tax. The overhead shows up in three places:

1. Startup time. JVM initialization adds 5-15 seconds before any work begins. Ray workers start in under a second.
2. Garbage collection. Large heap sizes trigger GC pauses that stall all threads in an executor. Ray's object store uses reference counting with no GC.
3. Python interop. PySpark routes Python UDFs through a JVM-to-Python bridge. Each row (or batch) crosses a serialization boundary. Ray workers are native Python processes, so your UDF runs directly.

For SQL-heavy batch analytics with few UDFs, Spark's JVM overhead is negligible. For Python-heavy ML pipelines with custom transforms on every row, the overhead compounds.

## When Python-native wins

Ray's architecture gives clear advantages when:

- Your workload is primarily Python functions applied to data
- You need GPU access (Ray workers are processes with direct GPU binding)
- You want to mix data processing with model training or serving in one cluster
- Low-latency per-task scheduling matters (Ray schedules a task in microseconds, Spark in milliseconds)
- You are building heterogeneous pipelines with different resource requirements per stage

## When Spark's design wins

Spark's architecture gives clear advantages when:

- You need SQL-first analytics with complex joins and aggregations
- Your organization has existing Hive metastore / data catalog integrations
- You need mature data governance (Delta Lake, Iceberg, table ACLs)
- Your team includes analysts who write SQL, not Python
- Fault tolerance at the stage level matters for very long-running jobs
- You are processing multi-petabyte datasets where Catalyst's optimization rules save hours of compute

Neither architecture is universally better. They optimize for different workloads, and understanding the tradeoffs helps you pick the right tool or combine both.
