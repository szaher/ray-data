## When to migrate, and when not to

Before migrating from Spark to Ray Data, be honest about whether it makes sense. Migrate when:

- Your pipeline is primarily Python UDFs applied to data, and Spark's JVM bridge is a bottleneck
- You want to integrate data processing with ML training/serving in a single cluster
- You need GPU-accelerated transforms (image processing, model inference)
- You are building online/streaming ML feature pipelines

Do **not** migrate when:

- Your workload is SQL-heavy with complex joins, window functions, and aggregations
- You rely on Delta Lake, Iceberg, or Hive metastore integration
- Your team's expertise is SQL, not Python
- Spark's performance is already adequate for your use case

| Spark Concept | Ray Data Equivalent | Notes |
|---|---|---|
| `SparkSession` | `ray.init()` | Ray has no session object; init starts the cluster |
| `DataFrame` | `Dataset` | Both are distributed collections of rows/records |
| RDD Partition | Block | A block is an Arrow table or Pandas DataFrame chunk |
| Executor | Worker | Ray workers are Python processes, not JVMs |
| `broadcast()` | `ray.put()` | Shared-memory object store replaces broadcast protocol |
| Spark UDF | `map` / `map_batches` | No UDF registration needed, just Python functions |
| Catalyst | (none) | No general-purpose optimizer; you optimize manually |
| `cache()` / `persist()` | `materialize()` | Forces execution and stores results |
| Spark SQL | (none) | No SQL interface; use functional API |
| `repartition(n)` | `repartition(n)` | Same concept, different implementation |

## Pattern 1: simple read-transform-write

This is the easiest migration pattern. The pipeline reads data, applies transformations, and writes output.

<CodeBlock language="python" filename="spark_etl.py" code={`# Spark version
from pyspark.sql import SparkSession
from pyspark.sql.functions import col, lower, trim, when
spark = SparkSession.builder.appName("etl").getOrCreate()
df = spark.read.parquet("s3://bucket/raw/")
result = (
    df.filter(col("status") == "active")
    .withColumn("email", lower(trim(col("email"))))
    .withColumn("tier", when(col("spend") > 1000, "gold")
                        .when(col("spend") > 100, "silver")
                        .otherwise("bronze"))
    .select("user_id", "email", "tier", "spend")
)
result.write.parquet("s3://bucket/processed/", mode="overwrite")
`} />

<CodeBlock language="python" filename="ray_data_etl.py" code={`# Ray Data version
import ray
ray.init()
def transform(batch):
    # Filter active users
    batch = batch[batch["status"] == "active"]
    # Normalize email
    batch["email"] = batch["email"].str.lower().str.strip()
    # Assign tier
    batch["tier"] = "bronze"
    batch.loc[batch["spend"] > 100, "tier"] = "silver"
    batch.loc[batch["spend"] > 1000, "tier"] = "gold"
    # Select columns
    return batch[["user_id", "email", "tier", "spend"]]
ds = ray.data.read_parquet("s3://bucket/raw/")
result = ds.map_batches(transform, batch_format="pandas")
result.write_parquet("s3://bucket/processed/")
`} />

The Spark version uses column expressions; the Ray Data version uses Pandas operations inside `map_batches`. Both are vectorized. Spark benefits from predicate pushdown on the `status` filter. Ray Data reads all data then filters in Python.

## Pattern 2: broadcast variable to ray.put()

<CodeBlock language="python" filename="spark_broadcast.py" code={`# Spark: broadcast a lookup table
from pyspark.sql.functions import col
lookup_data = {"US": "United States", "UK": "United Kingdom", "DE": "Germany"}
broadcast_lookup = spark.sparkContext.broadcast(lookup_data)
@udf(returnType=StringType())
def expand_country(code):
    return broadcast_lookup.value.get(code, code)
result = df.withColumn("country_name", expand_country(col("country_code")))
`} />

<CodeBlock language="python" filename="ray_data_broadcast.py" code={`# Ray Data: use ray.put() for shared data
import ray
lookup_data = {"US": "United States", "UK": "United Kingdom", "DE": "Germany"}
lookup_ref = ray.put(lookup_data)
def expand_countries(batch):
    lookup = ray.get(lookup_ref)
    batch["country_name"] = batch["country_code"].map(
        lambda code: lookup.get(code, code)
    )
    return batch
result = ds.map_batches(expand_countries, batch_format="pandas")
`} />

`ray.put()` stores the lookup in the shared-memory object store. Workers on the same node access it via zero-copy memory mapping. Unlike Spark's broadcast, there is no explicit "destroy" call. Ray handles cleanup via reference counting.

## Pattern 3: Spark SQL to functional API

This pattern requires the most thought. You cannot just port SQL. You need to rethink the logic.

<CodeBlock language="python" filename="spark_sql.py" code={`# Spark: SQL with window functions
df.createOrReplaceTempView("transactions")
result = spark.sql("""
    SELECT
        user_id,
        amount,
        transaction_date,
        SUM(amount) OVER (
            PARTITION BY user_id
            ORDER BY transaction_date
            ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
        ) as running_total,
        ROW_NUMBER() OVER (
            PARTITION BY user_id
            ORDER BY amount DESC
        ) as rank
    FROM transactions
    WHERE amount > 0
""")
`} />

<CodeBlock language="python" filename="ray_data_equivalent.py" code={`# Ray Data: rewrite as grouped operations
# Option 1: Use groupby + map_groups for complex per-group logic
def compute_running_totals(group):
    group = group[group["amount"] > 0]
    group = group.sort_values("transaction_date")
    group["running_total"] = group["amount"].cumsum()
    group["rank"] = group["amount"].rank(method="first", ascending=False).astype(int)
    return group
result = ds.groupby("user_id").map_groups(
    compute_running_totals,
    batch_format="pandas",
)
# Option 2: If the dataset fits, use DuckDB as an intermediate
# pip install duckdb
import duckdb
def sql_transform(batch):
    return duckdb.sql("""
        SELECT *, SUM(amount) OVER (
            ORDER BY transaction_date
            ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
        ) as running_total
        FROM batch
        WHERE amount > 0
    """).df()
result = ds.map_batches(sql_transform, batch_format="pandas")
`} />

Window functions are one of the hardest patterns to migrate. `map_groups` gives you per-group Pandas DataFrames where you can use any Pandas operation. For SQL-heavy logic, embedding DuckDB inside `map_batches` is a pragmatic bridge: you get SQL syntax with Ray's distributed execution.

## Pattern 4: Spark caching to Ray materialization

<CodeBlock language="python" filename="spark_cache.py" code={`# Spark: cache intermediate results
processed = df.filter(...).withColumn(...)
processed.cache()  # Store in executor memory
# Reuse the cached DataFrame
count = processed.count()
sample = processed.limit(100).toPandas()
aggregated = processed.groupBy("city").count()
processed.unpersist()  # Release cache
`} />

<CodeBlock language="python" filename="ray_data_materialize.py" code={`# Ray Data: materialize to force execution
processed = ds.filter(...).map_batches(...)
materialized = processed.materialize()  # Execute and store in object store
# Reuse the materialized dataset
count = materialized.count()
sample = materialized.take(100)
aggregated = materialized.groupby("city").aggregate(Count())
# No explicit unpersist — reference counting handles cleanup
`} />

## Common gotchas

### 1. Lazy vs streaming evaluation

Spark is lazy: nothing executes until an action (`count()`, `collect()`, `write`). Ray Data is also lazy for operator construction, but execution is streaming. Blocks flow through the pipeline as soon as they are ready. Ray Data may start processing before all data is read, which is usually good, but errors can appear at different points in the pipeline than you expect.

### 2. Partitioning semantics

<CodeBlock language="python" filename="partitioning_gotcha.py" code={`# Spark: partitions are tightly controlled
df.repartition(100, "user_id")  # Hash-partition by user_id into 100 partitions
# All rows with the same user_id end up in the same partition
# Ray Data: blocks are not hash-partitioned by default
ds.repartition(100)  # Random redistribution into 100 blocks
# Rows with the same user_id may be in different blocks
# For grouped operations, use groupby which handles this internally
`} />

### 3. Error handling

<CodeBlock language="python" filename="error_handling.py" code={`# Spark: errors surface at action time
# Writing a UDF that might fail
@udf(returnType=StringType())
def risky_udf(value):
    return str(int(value))  # Throws on non-numeric
# Error appears when you call .collect() or .write
# Ray Data: errors surface per-block during streaming
def risky_transform(row):
    row["value"] = int(row["raw_value"])  # Throws on non-numeric
    return row
# Wrap in try/except for robustness
def safe_transform(row):
    try:
        row["value"] = int(row["raw_value"])
    except (ValueError, TypeError):
        row["value"] = None
    return row
result = ds.map(safe_transform)
`} />

### 4. No schema enforcement

Spark DataFrames have strict schemas. If a column is declared as `IntegerType`, Spark enforces it. Ray Data blocks are Arrow tables with inferred schemas. If your `map_batches` function returns a batch with different columns or types, you may get silent schema drift across blocks. Always validate your output schema.

## What changes, what stays

What stays the same: reading/writing Parquet and CSV, the concept of lazy evaluation, map and filter operations, GroupBy aggregations, repartitioning, the general pipeline pattern of read-transform-write.

What changes: no SQL, no Catalyst optimizer, no complex joins (or manual implementation), no schema enforcement, no partitioned writes, different error surfaces, different partitioning semantics, different caching semantics, no Hive metastore integration.

The migration is not a mechanical find-and-replace. It requires rethinking how you express your data logic, trading Spark's declarative SQL-centric model for Ray Data's functional Python-centric model. The payoff is a Python-native runtime with direct access to the ML ecosystem.
