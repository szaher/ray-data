## Running Spark SQL and Ray ML on one cluster

You do not have to choose between Spark and Ray. You can run Spark SQL for the parts of your pipeline where it excels and Ray for ML training and serving, on the same cluster, passing data between them without writing to disk.

**RayDP** (Ray Data Processing) is an open-source library that makes this possible. It runs Apache Spark on a Ray cluster by implementing Spark executors as Ray actors.

## What RayDP does

RayDP is a thin integration layer that:

1. Starts a Spark session that uses Ray as its cluster manager
2. Launches Spark executors as Ray actors (not as standalone JVM processes)
3. Enables zero-copy conversion between Spark DataFrames and Ray Datasets
4. Shares cluster resources between Spark and Ray workloads

## Installation

<CodeBlock language="bash" filename="install.sh" code={`# Install RayDP (requires Ray and PySpark)
pip install raydp
# Verify installation
python -c "import raydp; print(raydp.__version__)"
`} />

RayDP requires a working Ray installation and a compatible PySpark version. Check the RayDP release notes for the supported PySpark range — it does not require a standalone Spark cluster. Ray provides the cluster.

## Setting up a Spark session on Ray

<CodeBlock language="python" filename="raydp_setup.py" code={`import ray
import raydp
# Step 1: Initialize Ray
ray.init()
# Step 2: Create a Spark session on Ray
spark = raydp.init_spark(
    app_name="spark_on_ray",
    num_executors=4,           # Number of Spark executors (Ray actors)
    executor_cores=4,          # CPU cores per executor
    executor_memory="8g",      # Memory per executor
    configs={
        "spark.sql.adaptive.enabled": "true",
        "spark.sql.shuffle.partitions": "200",
    },
)
# Now use 'spark' like any normal SparkSession
print(type(spark))  # <class 'pyspark.sql.session.SparkSession'>
# When done, stop the Spark session (releases Ray actors)
raydp.stop_spark()
ray.shutdown()
`} />

`raydp.init_spark()` creates a standard PySpark `SparkSession`, but instead of connecting to YARN or a standalone Spark master, it launches Spark executors as Ray actors. You get a fully functional SparkSession with the same API, the same Catalyst optimizer, and the same Spark SQL support.

## Running Spark SQL on Ray

Once you have the session, all standard PySpark operations work normally.

<CodeBlock language="python" filename="raydp_sql.py" code={`import ray
import raydp
ray.init()
spark = raydp.init_spark("sql_demo", num_executors=2,
                          executor_cores=2, executor_memory="4g")
# Read data with Spark
df = spark.read.parquet("s3://bucket/transactions/")
# Create a temp view and run SQL
df.createOrReplaceTempView("transactions")
result = spark.sql("""
    SELECT
        customer_id,
        COUNT(*) as num_transactions,
        SUM(amount) as total_spend,
        AVG(amount) as avg_spend,
        MAX(transaction_date) as last_transaction
    FROM transactions
    WHERE status = 'completed'
    GROUP BY customer_id
    HAVING COUNT(*) >= 5
    ORDER BY total_spend DESC
""")
result.show()
# All Spark features work: joins, window functions, UDFs, etc.
from pyspark.sql.window import Window
from pyspark.sql.functions import row_number, col
window = Window.partitionBy("customer_id").orderBy(col("amount").desc())
ranked = df.withColumn("rank", row_number().over(window))
`} />

## Converting between Spark DataFrames and Ray Datasets

This is where RayDP pays off. You can move data between Spark and Ray without writing to disk.

<CodeBlock language="python" filename="raydp_conversion.py" code={`import ray
import raydp
ray.init()
spark = raydp.init_spark("conversion_demo", num_executors=2,
                          executor_cores=2, executor_memory="4g")
# --- Spark to Ray Dataset ---
# Spark SQL for ETL
spark_df = spark.read.parquet("s3://bucket/raw_data/")
spark_df.createOrReplaceTempView("raw")
cleaned = spark.sql("""
    SELECT user_id, features, label
    FROM raw
    WHERE features IS NOT NULL AND label IS NOT NULL
""")
# Convert Spark DataFrame to Ray Dataset (in-memory transfer)
ray_ds = raydp.spark.spark_dataframe_to_ray_dataset(cleaned)
print(type(ray_ds))     # <class 'ray.data.dataset.Dataset'>
print(ray_ds.schema())  # Arrow schema
# --- Ray Dataset to Spark DataFrame ---
# After processing in Ray, convert back to Spark
processed_ds = ray_ds.map_batches(some_transform, batch_format="pandas")
spark_df_back = raydp.spark.ray_dataset_to_spark_dataframe(
    spark, processed_ds
)
spark_df_back.show()
`} />

The conversion uses Apache Arrow as the intermediate format. Data is transferred through Ray's object store, avoiding serialization to disk. For large datasets, this is significantly faster than writing Parquet as an intermediate format.

## End-to-end example: Spark ETL + Ray ML

This is the canonical RayDP use case: Spark for data preparation, Ray for model training.

<CodeBlock language="python" filename="raydp_e2e.py" code={`import ray
import raydp
from ray.train.torch import TorchTrainer
from ray.train import ScalingConfig
ray.init()
# --- Phase 1: Spark ETL ---
spark = raydp.init_spark("etl_ml_pipeline", num_executors=4,
                          executor_cores=4, executor_memory="8g")
# Complex SQL that Spark handles well
raw = spark.read.parquet("s3://datalake/events/")
users = spark.read.parquet("s3://datalake/users/")
raw.createOrReplaceTempView("events")
users.createOrReplaceTempView("users")
features = spark.sql("""
    SELECT
        u.user_id,
        u.age,
        u.signup_days_ago,
        e.event_count,
        e.avg_session_duration,
        e.distinct_pages,
        u.is_premium as label
    FROM users u
    JOIN (
        SELECT
            user_id,
            COUNT(*) as event_count,
            AVG(session_duration) as avg_session_duration,
            COUNT(DISTINCT page_id) as distinct_pages
        FROM events
        WHERE event_date >= '2024-01-01'
        GROUP BY user_id
    ) e ON u.user_id = e.user_id
    WHERE u.age IS NOT NULL
""")
# --- Phase 2: Convert to Ray Dataset ---
train_ds = raydp.spark.spark_dataframe_to_ray_dataset(features)
# Stop Spark to free cluster resources for training
raydp.stop_spark()
# --- Phase 3: Ray ML Training ---
import torch
import torch.nn as nn
def train_func(config):
    import ray.train
    train_ds = ray.train.get_dataset_shard("train")
    model = nn.Sequential(
        nn.Linear(5, 64), nn.ReLU(),
        nn.Linear(64, 32), nn.ReLU(),
        nn.Linear(32, 1), nn.Sigmoid(),
    )
    optimizer = torch.optim.Adam(model.parameters(), lr=0.001)
    loss_fn = nn.BCELoss()
    for epoch in range(10):
        for batch in train_ds.iter_torch_batches(batch_size=256):
            features = torch.stack([
                batch["age"], batch["signup_days_ago"],
                batch["event_count"], batch["avg_session_duration"],
                batch["distinct_pages"],
            ], dim=1).float()
            labels = batch["label"].float().unsqueeze(1)
            pred = model(features)
            loss = loss_fn(pred, labels)
            optimizer.zero_grad()
            loss.backward()
            optimizer.step()
        ray.train.report({"loss": loss.item()})
trainer = TorchTrainer(
    train_func,
    datasets={"train": train_ds},
    # gpu-example: spark-ray-train
    scaling_config=ScalingConfig(num_workers=4, use_gpu=True),
)
result = trainer.fit()
print(f"Final loss: {result.metrics['loss']:.4f}")
`} />

## How RayDP works internally

<MermaidDiagram chart={`graph TD
    subgraph "RayDP Internal Architecture"

        SB --> RA2[Ray Actor\nSpark Executor 2]
        SB --> RA3[Ray Actor\nSpark Executor 3]

RayDP implements a custom `ExternalClusterManager` for Spark. When Spark requests executors, RayDP creates Ray actors instead of launching standalone JVM processes. Each Ray actor runs a JVM inside it. Spark's execution engine (Catalyst, Tungsten, shuffle) works exactly as it does on YARN or Kubernetes, but resource allocation and scheduling are managed by Ray.

## When to use RayDP

Good fit:
- Existing Spark ETL pipelines that feed into Ray ML training
- Teams with Spark SQL expertise that want to add ML capabilities
- Workloads that genuinely need both SQL analytics and distributed ML
- Unified cluster management: one Ray cluster instead of separate Spark and ML clusters

Not ideal:
- Pure data processing with no ML component (just use Spark)
- Pure ML with no SQL requirements (just use Ray Data)
- Workloads where Spark's JVM memory overhead is unacceptable
- Environments where installing Java/JVM is not feasible

## Resource management tips

<CodeBlock language="python" filename="raydp_resources.py" code={`# Tip 1: Release Spark resources before ML training
spark = raydp.init_spark("pipeline", num_executors=8,
                          executor_cores=4, executor_memory="8g")
# ... do Spark ETL ...
ray_dataset = raydp.spark.spark_dataframe_to_ray_dataset(spark_df)
raydp.stop_spark()  # Frees 8 executors worth of resources
# Now all cluster resources are available for Ray training
# Tip 2: Size executors to leave room for Ray workers
# On a 64-core, 256GB cluster:
spark = raydp.init_spark("shared", num_executors=4,
                          executor_cores=4, executor_memory="16g")
# Uses 16 cores + 64GB for Spark, leaving 48 cores + 192GB for Ray
# Tip 3: Use dynamic allocation when available
spark = raydp.init_spark("dynamic", num_executors=2,
                          executor_cores=4, executor_memory="8g",
                          configs={
                              "spark.dynamicAllocation.enabled": "true",
                              "spark.dynamicAllocation.minExecutors": "1",
                              "spark.dynamicAllocation.maxExecutors": "8",
                          })
`} />

RayDP gives you full Spark SQL while keeping everything on a single Ray cluster. For organizations transitioning from Spark to Ray, it provides a practical bridge: start with RayDP for your existing ETL, gradually migrate simple pipelines to Ray Data, and keep Spark for the SQL-heavy parts where it genuinely excels.
