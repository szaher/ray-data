## Performance is not one-dimensional

Benchmarking Spark against Ray Data with a single number is misleading. Performance depends on workload shape, data size, cluster configuration, and what you are optimizing for: throughput, latency, resource efficiency, or developer productivity. This lesson gives you a framework for making the right choice.

## Data size regimes

### Small data (less than 10 GB)

At this scale, framework overhead dominates actual computation time.

Spark's JVM startup typically takes 5-15 seconds. Creating a SparkSession, launching executors, and initializing the Catalyst optimizer adds fixed overhead. For a 1 GB CSV read-filter-write pipeline, you might spend more time starting Spark than processing data.

Ray Data has Python-native startup (typically under 1 second for `ray.init()` on a local cluster). For small datasets, streaming execution avoids materializing the full dataset. A 1 GB pipeline can start producing output almost immediately.

For interactive work, prototyping, and small pipelines, Ray Data's lower startup overhead makes it feel significantly faster. For batch jobs where startup is amortized over hours of processing, this difference is irrelevant.

### Medium data (10-100 GB)

This is where the choice depends most on workload type.

Catalyst optimization shines here for Spark. A complex query with joins, filters, and aggregations benefits from predicate pushdown, join reordering, and column pruning. Spark can process 50 GB with a well-optimized plan faster than a naive implementation in any framework.

For map-heavy pipelines (apply a Python function to every row/batch), Ray Data avoids the JVM-Python serialization tax. A pipeline that runs a PyTorch model on every batch of images will be faster in Ray Data because the model runs directly in the worker process.

SQL-shaped workloads favor Spark. Python-transform-shaped workloads favor Ray Data.

### Large data (more than 100 GB)

At this scale, both frameworks are effective, but different factors dominate.

Spark has a mature shuffle implementation, spill-to-disk handling, and Adaptive Query Execution (AQE) that make it robust for very large shuffles (joins, groupbys on high-cardinality keys). Decades of optimization on JVM garbage collection, Tungsten memory management, and code generation make Spark competitive even for Python workloads via Pandas UDFs.

Ray Data's streaming execution keeps memory footprint low by processing blocks incrementally. For read-transform-write pipelines without shuffles, Ray Data can process datasets much larger than available memory. For workloads with large shuffles, Ray Data is less mature than Spark.

For shuffle-heavy workloads (big joins, full sorts), Spark is more mature. For streaming map workloads (ETL, inference), Ray Data's memory efficiency is an advantage.

### Serialization cost

Traditional scalar Python UDFs in Spark use cloudpickle by default unless Arrow optimization is enabled; Pandas UDFs use Arrow for efficient batched transfer. Either way, data crosses the JVM-Python boundary, and for pipelines with many Python UDFs, this cost compounds.

Ray Data avoids the JVM-Python bridge entirely. Data is serialized into and out of the shared-memory object store in Arrow format, and Python workers read it directly. Serialization still happens when data moves between nodes, but there is no cross-language boundary within a single node.

### JVM garbage collection

Spark executors are long-running JVM processes. As heap usage grows, the JVM garbage collector runs, and full GC pauses can stall an executor for seconds. This creates tail-latency issues: a single executor's GC pause can delay an entire stage. Tuning GC is an art (`-XX:+UseG1GC`, `-XX:ParallelGCThreads`, heap sizing).

Ray workers are long-lived Python processes. Python has its own garbage collector, but it does not cause the same stop-the-world pauses as JVM full GC. The object store uses reference counting and memory-mapped files, keeping large data out of the Python heap.

### Task scheduling latency

Spark schedules tasks at the stage level. A stage may contain thousands of tasks, but they are all planned at once. Individual task scheduling overhead is low (milliseconds).

Ray typically schedules individual tasks with microsecond-level latency. This matters for fine-grained task graphs (many small tasks), but for bulk data processing with large blocks, both are fast enough.

## Spark strengths: where it wins

1. **SQL and complex analytics.** If your workload is SQL (joins, window functions, CTEs, subqueries), Spark is the clear winner. Catalyst optimizes these automatically. Ray Data has no SQL engine.

2. **Data governance and catalogs.** Delta Lake, Apache Iceberg, Apache Hudi, Unity Catalog, Hive Metastore. The data lake ecosystem is built around Spark. ACID transactions, schema evolution, time travel, table access control are all mature in Spark and absent in Ray Data.

3. **Ecosystem breadth.** Hundreds of data source connectors (JDBC, Kafka, Cassandra, MongoDB, Elasticsearch, Snowflake). Structured Streaming for event processing. MLlib for distributed ML (though less used now). GraphX for graph processing.

4. **Mature shuffle.** For workloads that require large shuffles (joining two 100 GB tables on a high-cardinality key), Spark's external shuffle service, sort-merge join, and spill-to-disk handling are battle-tested at petabyte scale.

5. **Managed platforms.** Databricks, Amazon EMR, Google Dataproc, Azure Synapse offer production-grade managed Spark with monitoring, auto-scaling, and enterprise support.

## Ray Data strengths: where it wins

1. **Python-native ML integration.** Ray Data feeds directly into Ray Train, Ray Tune, and Ray Serve. Data processing, training, hyperparameter tuning, and model serving share one cluster and one API surface. No Parquet files needed as intermediate hand-off.

2. **GPU workloads.** Ray workers bind directly to GPUs. Running a PyTorch model inside `map_batches` with `num_gpus=1` just works. Spark supports executor- and task-level GPU resource requests via `ResourceProfiles` since Spark 3.1, but the integration adds configuration complexity compared to Ray's native approach.

3. **Heterogeneous resources.** Different stages of a Ray Data pipeline can request different resources. Image decoding needs CPUs, model inference needs GPUs, post-processing needs CPUs again. Ray handles this naturally. Spark supports `ResourceProfiles` and Dynamic Resource Allocation, but mixing resource types across stages requires more explicit configuration.

4. **Streaming execution.** Ray Data processes blocks through the pipeline incrementally. A 500 GB read-transform-write pipeline may only need 10 GB of memory because it never materializes the full dataset at once. Spark stages force materialization at shuffle boundaries.

5. **Low overhead for Python.** No JVM, no JVM-Python serialization bridge, and no JVM full-GC pauses. For workloads that are fundamentally Python functions on data, Ray Data removes the cross-language overhead layer.

## The hybrid approach

This pattern is increasingly common in production ML systems:

1. Spark handles data ingestion, SQL-heavy ETL, complex joins across many tables, data quality checks, and writing to a governed data lake (Delta Lake or Iceberg)
2. A data lake (Parquet on S3/GCS) serves as the hand-off point
3. Ray Data reads the prepared features, applies Python-heavy preprocessing (tokenization, image augmentation, feature engineering), and feeds into Ray Train for distributed model training

### Choose Spark when:
- Your workload is primarily SQL with complex joins and aggregations
- You need data governance: ACID transactions, schema enforcement, audit logs
- Your team includes analysts and data engineers who write SQL
- You need connectors to enterprise data sources (JDBC, Kafka, Snowflake)
- You are operating at petabyte scale with heavy shuffles
- You need a managed platform with enterprise support (Databricks)

### Choose Ray Data when:
- Your workload is Python functions applied to data (transforms, inference)
- You need GPU-accelerated processing in your data pipeline
- You are building an end-to-end ML pipeline (data prep + training + serving)
- You want to minimize memory footprint with streaming execution
- Your transforms are heterogeneous (different resources per stage)
- You are already using Ray for training or serving

### Choose both when:
- You have existing Spark ETL and want to add ML capabilities
- Your pipeline has a clear SQL-heavy phase followed by a Python-heavy phase
- Different teams own different parts of the pipeline (data eng vs ML eng)
- You want Spark's governance for your data lake and Ray's efficiency for ML

## Cost considerations

Beyond raw performance, consider operational costs.

Running separate Spark and Ray clusters means duplicate infrastructure. RayDP or a hybrid approach reduces this.

Developer productivity matters too. Spark UDF debugging is painful (JVM stack traces, serialization errors). Ray Data debugging is standard Python with standard tracebacks. This affects development velocity.

Spark executors hold fixed resources for the duration of a job. Ray's task-based model can dynamically reallocate resources. For bursty workloads, Ray can be more resource-efficient.

On ecosystem lock-in: Spark runs on any Hadoop-compatible cluster and has multiple managed options. Ray's managed options are growing (Anyscale) but the ecosystem is younger.

The best choice depends on your specific workload, team, and infrastructure. Both frameworks are excellent. They just optimize for different things. Understanding their strengths lets you pick the right tool, or combine them, for your situation.
