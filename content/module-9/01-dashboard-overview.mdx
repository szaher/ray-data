## What is the Ray Dashboard?

The Ray Dashboard is a web-based UI that shows you what your Ray cluster is doing in real time. It runs at **http://127.0.0.1:8265** by default and covers jobs, actors, logs, and resource utilization from one place.

No extra installation required. The dashboard ships with Ray and starts automatically.

## Launching the dashboard

The dashboard starts whenever a Ray cluster starts. There are two common paths:

<CodeBlock language="python" code={`import ray
# Option 1: ray.init() starts a local cluster + dashboard
context = ray.init()
print(context.dashboard_url)
# Output: "127.0.0.1:8265"
# Option 2: With a custom dashboard port
context = ray.init(dashboard_port=9999)
print(context.dashboard_url)
# Output: "127.0.0.1:9999"`} language="python" filename="launch_dashboard.py" />

When deploying a multi-node cluster, you typically start the head node from the command line and bind the dashboard to all network interfaces so remote users can reach it:

<CodeBlock code={`# Start the head node with the dashboard accessible externally
ray start --head --dashboard-host=0.0.0.0 --dashboard-port=8265
# Workers connect to the head node
ray start --address=<head-node-ip>:6379`} language="bash" filename="cluster_start.sh" />

Once the cluster is running, open your browser to the head node's IP on port 8265.

## How the dashboard works

The dashboard is a lightweight web server on the head node. It does not run your workloads. It observes them by querying the **Global Control Store (GCS)**, which holds all cluster metadata.

Every raylet on every worker node continuously reports its status to the GCS: CPU usage, memory, running tasks, actor states. The dashboard reads this data and renders it in real time.

## Dashboard tabs

The navigation bar across the top has six main tabs.

**Jobs** shows every Ray job (submitted via `ray.init()`, `ray job submit`, or the Jobs API). Each job displays a progress bar, the number of tasks completed, and any error traces. Click into a job to see its individual tasks, their execution times, and their logs. For Ray Data pipelines, this tab also shows the operator-level execution graph.

**Actors** lists every actor in the cluster: class names, current state (ALIVE, DEAD, RESTARTING), the node each is running on, restart counts, and resource consumption. Useful for debugging long-running services and stateful computations.

**Metrics** displays time-series charts for CPU, GPU, memory, disk, and network utilization across the cluster. Ray exports Prometheus-compatible metrics, and this tab renders a built-in Grafana-style view with per-node breakdowns. You can zoom into specific time windows.

**Logs** gives you access to stdout and stderr for every task, actor, and system process in the cluster. You can filter by node, job, or actor. No more SSH-ing into machines to tail log files.

**Cluster** shows cluster topology: how many nodes are up, their IP addresses, CPU/GPU/memory/disk resources, and the number of workers on each node. If a node goes down, it appears as DEAD with a timestamp.

**Serve** is dedicated to Ray Serve deployments. It shows deployment status, replica counts, request latency, and error rates. This tab only appears when Ray Serve is active.

## The Jobs page in detail

The Jobs page is where you spend most of your time monitoring Ray Data pipelines. For each job you see:

- **Job ID** and **submission ID** as unique identifiers
- **Status**: PENDING, RUNNING, SUCCEEDED, or FAILED
- **Progress bar** showing a visual breakdown of tasks by state (scheduled, running, finished, failed)
- **Duration** as wall-clock time since the job started
- **Error message** with the full Python traceback if the job failed

Clicking into a running job reveals the **Task Timeline**, a Gantt-chart view showing when each task started, how long it ran, and on which node. For Ray Data jobs, you also see the **operator graph**: the DAG of map, filter, and other operators with per-operator progress.

## The Cluster page in detail

The Cluster page gives you hardware-level visibility:

- **Node table**: each row is a physical or virtual machine in the cluster, showing its IP, hostname, CPU count, GPU count, total memory, disk, and current utilization percentages
- **Status indicators**: green for ALIVE, red for DEAD, yellow for draining
- **Resource bars**: colored bars showing how much CPU, memory, GPU, and object store memory is in use versus available

This page helps you answer questions like "Are all my nodes actually being used?" and "Is one node overloaded while others are idle?"

## Accessing the dashboard programmatically

The dashboard also exposes a REST API. You can query it from scripts or monitoring tools:

<CodeBlock code={`import requests
# Get cluster status
resp = requests.get("http://127.0.0.1:8265/api/cluster_status")
print(resp.json())
# List all jobs
resp = requests.get("http://127.0.0.1:8265/api/jobs/")
for job in resp.json():
    print(f"{job['job_id']}: {job['status']}")
# Check node information
resp = requests.get("http://127.0.0.1:8265/nodes?view=summary")
print(resp.json())`} language="python" filename="dashboard_api.py" />

## Dashboard configuration options

You can customize the dashboard behavior when starting Ray:

<CodeBlock code={`import ray
ray.init(
    dashboard_port=8265,            # Custom port
    include_dashboard=True,         # Enabled by default
    _metrics_export_port=8080,      # Prometheus metrics port
)
# To disable the dashboard entirely (saves some resources):
ray.init(include_dashboard=False)`} language="python" filename="dashboard_config.py" />

For production clusters, always set `--dashboard-host=0.0.0.0` so the dashboard is accessible from outside the head node. Pair this with network policies or a reverse proxy to control access.
