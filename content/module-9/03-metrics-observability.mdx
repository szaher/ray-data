## Ray's built-in metrics endpoint

Ray exposes a Prometheus-compatible metrics endpoint on every node. By default, this endpoint lives at port **8080** and serves metrics in the standard Prometheus text format.

These metrics are collected automatically. You do not need to instrument your code to get cluster-level and Ray Data-level metrics. They cover CPU, memory, GPU, object store, task counts, and Ray Data operator statistics.

<CodeBlock code={`# Fetch raw Prometheus metrics from the head node
curl http://127.0.0.1:8080/metrics
# You'll see output like:
# ray_data_output_rows{dataset="...",operator="..."} 12000000
# ray_data_spilled_bytes{} 0
# ray_object_store_memory_used_bytes{} 536870912
# ray_node_cpu_utilization{} 78.5
# ray_component_num_fds{component="raylet"} 42`} language="bash" filename="fetch_metrics.sh" />

## Key Ray Data metrics

These are the most important Prometheus metrics for monitoring data pipelines:

| Metric | Description |
|--------|------------|
| `ray_data_output_rows` | Total rows produced by each operator |
| `ray_data_spilled_bytes` | Bytes spilled to disk when object store is full |
| `ray_data_current_ops` | Number of operators currently executing |
| `ray_data_iterator_total_blocked_time_s` | Time the consumer spent waiting for data |
| `ray_data_iterator_total_time_s` | Total wall time of the data iterator |
| `ray_object_store_memory_used_bytes` | Object store memory currently in use |
| `ray_node_cpu_utilization` | Per-node CPU utilization percentage |
| `ray_node_mem_used` | Per-node system memory used in bytes |
| `ray_node_gpus_utilization` | Per-node GPU utilization percentage |

When `ray_data_spilled_bytes` starts climbing, your pipeline is running out of object store memory and spilling blocks to disk. This is a major performance concern. When `ray_data_iterator_total_blocked_time_s` is high relative to total time, your consumer is starving because upstream operators cannot produce data fast enough.

          - 'ray-head:8080'

## Structured logging

Ray provides a structured logging system that you can use inside tasks and actors. Logs are automatically collected and viewable in the Dashboard's Logs tab.

<CodeBlock code={`import ray
import logging
# Get a Ray-aware logger
logger = logging.getLogger("ray.data")
@ray.remote
def process_batch(batch):
    logger = logging.getLogger("ray.data")
    logger.info(f"Processing batch with {len(batch)} rows")
    try:
        result = transform(batch)
        logger.info(f"Batch processed successfully: {len(result)} output rows")
        return result
    except Exception as e:
        logger.error(f"Batch processing failed: {e}", exc_info=True)
        raise
# Configure log level for Ray Data
logging.getLogger("ray.data").setLevel(logging.DEBUG)
# Now all Ray Data internal logs will be visible:
# DEBUG: Starting operator Map(preprocess)
# DEBUG: Block 3 produced 50000 rows in 1.2s
# INFO: Operator Map(preprocess) completed: 24 blocks, 1200000 rows`} language="python" filename="structured_logging.py" />

Logs from tasks and actors are written to files on the node where they execute. The Dashboard aggregates these logs so you can view them from one location, without SSH-ing into individual machines.

## Custom metrics

When the built-in metrics are not enough, you can define your own using `ray.util.metrics`. Ray supports three metric types: **Counter**, **Gauge**, and **Histogram**.

<CodeBlock code={`import ray
from ray.util.metrics import Counter, Gauge, Histogram
# Define custom metrics (do this at module level or in actor __init__)
rows_processed = Counter(
    "my_pipeline_rows_processed",
    description="Total rows processed by my pipeline",
    tag_keys=("operator", "status"),
)
batch_latency = Histogram(
    "my_pipeline_batch_latency_seconds",
    description="Time to process each batch",
    boundaries=[0.1, 0.5, 1.0, 2.0, 5.0, 10.0],
    tag_keys=("operator",),
)
queue_depth = Gauge(
    "my_pipeline_queue_depth",
    description="Number of batches waiting to be processed",
    tag_keys=("operator",),
)
@ray.remote
class BatchProcessor:
    def __init__(self):
        # Metrics are per-process, so define them in __init__ for actors
        self.rows_counter = Counter(
            "actor_rows_processed",
            description="Rows processed by this actor",
        )
        self.latency = Histogram(
            "actor_batch_latency_s",
            description="Batch processing latency",
            boundaries=[0.01, 0.05, 0.1, 0.5, 1.0],
        )
    def process(self, batch):
        import time
        start = time.time()
        result = [transform(row) for row in batch]
        elapsed = time.time() - start
        self.rows_counter.inc(len(result))
        self.latency.observe(elapsed)
        return result`} language="python" filename="custom_metrics.py" />

Custom metrics appear alongside built-in metrics at the `/metrics` endpoint and can be scraped by Prometheus, charted in Grafana, and used to trigger alerts.

**Layer 1, Application.** Your pipeline code emits custom metrics for business-level KPIs (rows processed, error rates, latency) and structured logs for debugging.

**Layer 2, Ray.** The framework automatically collects system metrics (CPU, memory, GPU, object store) and aggregates logs. The dashboard provides real-time visibility.

**Layer 3, Infrastructure.** Prometheus scrapes all metrics for long-term storage and alerting. Grafana provides dashboards. Log storage (Elasticsearch, Loki) enables searching across historical logs.
