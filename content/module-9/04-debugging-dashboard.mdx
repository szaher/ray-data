## A systematic approach to debugging

When a Ray Data pipeline fails, hangs, or performs poorly, the dashboard is your primary diagnostic tool. Rather than guessing, follow a systematic workflow: observe the symptom, go to the right tab, form a diagnosis, apply a fix.

## Diagnosing out-of-memory (OOM)

OOM is the most common failure mode for data pipelines. The dashboard gives you multiple signals before a hard crash occurs.

**Step 1: Check the Metrics tab.** Look at the object store memory graph. If it is climbing toward the limit (default is 30% of system memory), your pipeline is producing blocks faster than it can consume them. The `ray_data_spilled_bytes` metric tells you if spilling has started. Once spilling begins, performance degrades significantly.

**Step 2: Check the Cluster tab.** Look at the system memory column for each node. If any node shows memory usage above 90%, that node is at risk of OOM. The operating system may kill Ray worker processes, which appear as DEAD workers.

**Step 3: Check the Jobs tab.** If a task was OOM-killed, the job will show a failed task with an error message like `Worker killed by OOM killer` or `RayOutOfMemoryError`. Click into the failed task to see which operator and which block triggered the failure.

Common fixes for OOM:

<CodeBlock code={`import ray
# Fix 1: Reduce block size so fewer bytes are in flight
ds = ray.data.read_parquet("s3://bucket/data/", override_num_blocks=200)
# Fix 2: Limit concurrent tasks to reduce memory pressure
ds = ds.map(heavy_transform, concurrency=4)
# Fix 3: Enable more aggressive backpressure
ctx = ray.data.DataContext.get_current()
ctx.target_max_block_size = 64 * 1024 * 1024  # 64MB blocks
# Fix 4: Increase object store memory (at ray.init time)
ray.init(
    _system_config={
        "object_store_memory": 10 * 1024 * 1024 * 1024,  # 10GB
    }
)
# Fix 5: Materialize and consume incrementally instead of all at once
for batch in ds.iter_batches(batch_size=1024):
    process_and_save(batch)`} language="python" filename="fix_oom.py" />

## Finding straggler tasks

Stragglers are tasks that take much longer than their peers, holding up the entire pipeline. They are easy to spot in the dashboard.

**Step 1: Open the Jobs tab** and click into your running job. Switch to the **Task Timeline** view.

**Step 2: Look for outliers.** In the Gantt chart, most tasks for an operator should have similar bar lengths. A straggler appears as one or two bars that extend far beyond the rest. Hover over the long bar to see:
- Task ID
- Duration (compare to the median)
- Node IP (is it always the same node?)
- Input block size (was this block much larger?)

**Step 3: Determine the cause.**

<CodeBlock code={`import ray
# Straggler cause 1: Data skew — some blocks have more data
# Diagnosis: Check block sizes in stats()
ds = ray.data.read_parquet("s3://bucket/skewed-data/")
result = ds.map(transform).materialize()
print(result.stats())
# Look for: "Block sizes: min=1MB, max=800MB, mean=50MB"
# The 800MB block is 16x larger than average — that's your straggler
# Fix: Repartition to equalize block sizes
ds = ray.data.read_parquet("s3://bucket/skewed-data/")
ds = ds.repartition(100)  # Redistribute into equal blocks
result = ds.map(transform).materialize()
# Straggler cause 2: Slow node — one machine has degraded hardware
# Diagnosis: In the timeline, stragglers always run on the same node IP
# Fix: Drain the node or enable speculative execution
# Straggler cause 3: Noisy neighbor — another process using resources
# Diagnosis: Check Cluster tab for the straggler's node
# Fix: Isolate Ray workers with cgroups or dedicated nodes`} language="python" filename="fix_stragglers.py" />

## Reading error traces

When a job fails, the Jobs tab shows the complete Python traceback. Here is how to get the most out of it.

**Step 1: Open the Jobs tab** and find the failed job (status: FAILED, shown in red).

**Step 2: Click the job** to expand its detail view. The error trace appears at the bottom. It shows:
- The Python exception type and message
- The full stack trace from your UDF code
- The Ray wrapper that shows which task, operator, and block failed

**Step 3: Trace back to your code.** The stack trace includes your function name and line number. Common patterns:

<CodeBlock code={`# Pattern 1: KeyError in a map function
# Error: KeyError: 'timestamp'
# Diagnosis: Some rows don't have the expected column
ds = ds.map(lambda row: {
    **row,
    "hour": row["timestamp"].hour,  # Fails if "timestamp" is missing
})
# Fix: Add defensive checks
ds = ds.map(lambda row: {
    **row,
    "hour": row["timestamp"].hour if "timestamp" in row else None,
})
# Pattern 2: TypeError from mismatched types
# Error: TypeError: unsupported operand type(s) for +: 'NoneType' and 'int'
# Diagnosis: A column contains None values
ds = ds.map(lambda row: {
    **row,
    "total": row["price"] + row["tax"],  # Fails if price or tax is None
})
# Fix: Handle None values
ds = ds.map(lambda row: {
    **row,
    "total": (row["price"] or 0) + (row["tax"] or 0),
})
# Pattern 3: Import error in a remote task
# Error: ModuleNotFoundError: No module named 'transformers'
# Diagnosis: The worker node doesn't have the required package
# Fix: Specify runtime environment
ray.init(runtime_env={"pip": ["transformers"]})`} language="python" filename="common_errors.py" />

## Log viewing

The Dashboard's Logs tab gives you access to stdout and stderr from every process in the cluster without SSH access.

**Navigation.** The Logs tab has a file browser on the left showing log files organized by node. Each node has:
- `raylet.out` / `raylet.err` for the raylet scheduler logs
- `gcs_server.out` / `gcs_server.err` for GCS logs (head node only)
- `worker-*.out` / `worker-*.err`, one file per worker process
- `dashboard.log` for dashboard server logs

**Finding task logs.** When you click into a specific task from the Jobs tab, a "View Logs" link takes you directly to the worker log file that contains that task's output. Any `print()` statements in your UDF appear here.

<CodeBlock code={`import ray
# Everything printed in a task appears in the Logs tab

    logger.info(f"Processing batch: {len(batch)} rows")

    return transform(batch)`} language="python" filename="debug_logging.py" />

## Actor debugging

Actors are long-running stateful objects that can fail, restart, and accumulate state over time. The Actors tab is built for debugging them.

What you see for each actor:
- **Class name** and **actor ID**
- **State**: ALIVE, DEAD, or RESTARTING
- **Node IP**, which machine it runs on
- **Restart count**, how many times it has been restarted (if `max_restarts` is set)
- **Resource usage**, CPU and memory consumed by this actor
- **Pending calls**, number of method calls waiting in the actor's queue

Common actor problems:

**Actor keeps restarting.** A high restart count with ALIVE state means the actor crashes and auto-recovers. Check its logs for the crash cause. Each restart loses in-memory state unless you implement checkpointing.

**Actor is DEAD.** It crashed and was not configured to restart. Click the actor to see the death cause. Set `max_restarts` to enable automatic recovery.

**Actor queue growing.** A rising number of pending calls means the actor cannot process requests fast enough. Consider increasing `max_concurrency` or sharding across multiple actor replicas.

## Dashboard tab quick reference

| Symptom | Go To | Look For |
|---------|-------|----------|
| Job failed | Jobs tab | Error traceback, failing operator |
| Pipeline slow | Metrics tab | CPU/GPU utilization, spilling |
| Pipeline hanging | Cluster tab | Dead nodes, resource exhaustion |
| OOM crashes | Metrics tab | Object store memory, spill bytes |
| Straggler tasks | Jobs tab (timeline) | Outlier task durations |
| Actor crashes | Actors tab | Restart count, DEAD state |
| Missing output | Logs tab | Task stdout/stderr |
| Data quality | Jobs tab + stats() | Row counts per operator |
